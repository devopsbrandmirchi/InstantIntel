## 1. Enable pg_net

The trigger uses `net.http_post()` to call your Edge Function with `?action=process`. Without pg_net, that call never happens.

- **Where:** Supabase Dashboard → **Database** → **Extensions**
- **Action:** Find **pg_net** and enable it.
- **Or in SQL Editor:**  
  `CREATE EXTENSION IF NOT EXISTS pg_net;`

## 2. Create the log table (if missing)

`run_daily_sales_processing` writes to `sales_processing_log`. If the table doesn’t exist, the “process” step will fail.

Run in **SQL Editor**:

```sql
CREATE TABLE IF NOT EXISTS public.sales_processing_log (
  id              bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  run_date        date NOT NULL,
  customer_id     integer,
  customer_name   text,
  status          text NOT NULL,
  notes           text,
  started_at      timestamptz NOT NULL DEFAULT clock_timestamp(),
  ended_at        timestamptz,
  elapsed_seconds numeric(10,2),
  rows_inserted   integer,
  error_message   text
);
```

Optional view for checking latest run (drop first if the view already exists with different column order):

```sql
DROP VIEW IF EXISTS public.v_sales_processing_latest;
CREATE VIEW public.v_sales_processing_latest AS
  SELECT * FROM public.sales_processing_log
  WHERE run_date = (SELECT MAX(run_date) FROM public.sales_processing_log)
  ORDER BY id;
```

## 4. Ensure the trigger calls the Edge Function with `?action=process`

Your `trigger_daily_sales_processing` must use **pg_net** to POST to:

`<your_supabase_url>/functions/v1/daily-sales-processor-dealer-wise?action=process`

with header `Authorization: Bearer <service_role_key>`.

If you’re not sure, replace the function with this (run in **SQL Editor**). It returns `json` to match your current return type:

```sql
CREATE OR REPLACE FUNCTION public.trigger_daily_sales_processing(
  p_supabase_url text,
  p_service_key  text
)
RETURNS json
LANGUAGE plpgsql SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_request_id bigint;
BEGIN
  SELECT net.http_post(
    url     := rtrim(p_supabase_url, '/') || '/functions/v1/daily-sales-processor-dealer-wise?action=process',
    headers := json_build_object(
                 'Content-Type',  'application/json',
                 'Authorization', 'Bearer ' || p_service_key
               )::jsonb,
    body    := '{}'::jsonb,
    timeout_milliseconds := 600000
  ) INTO v_request_id;

  RETURN json_build_object(
    'status',      'triggered',
    'request_id',  v_request_id,
    'message',     'Processing started in background. Check sales_processing_log for progress.'
  );
END;
$$;
```

## Troubleshooting: Trigger succeeds but no new rows in sales_processing_log

If Postman returns success and a request_id (e.g. 28) but nothing appears in sales_processing_log, the background HTTP call from pg_net to your Edge Function with `?action=process` is likely failing.

**1. Check pg_net response for that request** (SQL Editor, use your request_id e.g. 28):

```sql
SELECT id, status_code, error_msg, timed_out, content, created
FROM net._http_response
WHERE id = 28
ORDER BY created DESC LIMIT 1;
```

- No row: wait a minute or run `SELECT * FROM net._http_response ORDER BY created DESC LIMIT 5;`
- status_code 401/403: auth failed — trigger must send the **service_role** key. Confirm the Edge Function passes `SUPABASE_SERVICE_ROLE_KEY` as `p_service_key`.
- **status_code 500 and content `{"error": "upstream request timeout"}`**: The background call to `?action=process` hit a timeout: `run_daily_sales_processing()` took longer than the Edge Function’s upstream (Supabase API) timeout, so the RPC was cut off and nothing was written to the log. **Fix:** The Edge Function was updated to run `run_daily_sales_processing()` **directly** in the same request (no pg_net callback). Redeploy the Edge Function from the repo; then call the same Postman URL (no `?action=process`). The request will take 1–2+ minutes and must finish within the Edge Function max duration (e.g. 150s–300s depending on plan). If it still times out, optimize the DB function or run it via pg_cron instead.
- status_code 404/500 or error_msg set: check Edge Function logs and/or call the function directly (step 3).
- timed_out = true: process ran longer than pg_net timeout.

**2. Edge Function logs:** Dashboard → Edge Functions → daily-sales-processor-dealer-wise → Logs. Look for the `?action=process` call and any error.

**3. Call ?action=process directly from Postman:**

- URL: `https://paifqtwkewuqszvqcuam.supabase.co/functions/v1/daily-sales-processor-dealer-wise?action=process`
- Header: `Authorization: Bearer <SUPABASE_SERVICE_ROLE_key>` (Dashboard → Settings → API → service_role).

If this returns 200 and `{"success": true}`, the log table should get new rows. If 500, the JSON error usually says why (e.g. missing table, RLS, error in run_daily_sales_processing).

**4. Edge Function must pass service role key:** `p_service_key` must be `Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")`, not the anon key, so pg_net’s request to `?action=process` is authorized.
